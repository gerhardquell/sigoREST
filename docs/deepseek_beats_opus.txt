The Failure Catalog
So why does the harness swing performance by 25 points? The answer turns out to be embarrassingly specific.

Researchers at the University of Washington built a formal taxonomy of how tool-augmented LLMs fail. Their ToolCritic framework identified just eight distinct failure categories.

Format violations — the model outputs valid JSON, but the schema doesn’t match what the tool expects
Parameter hallucination — inventing a parameter name that doesn’t exist in the tool’s definition
Incomplete tool calls — calling a tool but leaving out required fields
Tool selection errors — picking the wrong tool entirely
Ordering errors — calling tools in the wrong sequence when they depend on each other
Missing calls — failing to call a tool when one was needed
Redundant calls — calling the same tool twice for no reason
Context-boundary degradation — losing track of prior tool results deep into a long session

Here’s the thing that makes this fixable: categories 1 through 3 are pure formatting problems. The model’s reasoning about what to do is correct. The output shape is wrong. And those three categories account for the most frequent failures in open models like DeepSeek, Qwen, and Granite.

Ahmad Awais found the same thing independently. Across every model he tested in CommandCode.ai, the “tool calling failures” reduced to a handful of specific patterns:

Sending null for an optional field instead of simply omitting it from the JSON
Passing "foo" as a bare string where the schema expects ["foo"] (an array)
Returning "[x, y]" as a string-wrapped array instead of an actual [x, y] array
Using Write when the tool registry expects write (case sensitivity)
Four patterns. That’s the catalog for every open model he tested.

This is what Awais calls a “small, finite completion set.” The models aren’t failing randomly. They’re failing predictably, in the same ways, over and over. And predictable failures are fixable failures.

The fix doesn’t live in the model. It lives in the middleware between the model and the tools. That middleware is part of what the industry now calls the harness.

What’s Actually Inside a Harness
Martin Fowler put it simply: Agent = Model + Harness. The harness is everything that isn’t the model weights. System prompts, tool definitions, middleware, memory, sub-agent configs. It’s the code that decides what the model sees, what tools it can reach, and what happens to its output before anything executes.

The AHE paper from Fudan and Peking University broke this down into seven editable components and measured what each one contributes when improved in isolation. The results were uneven in a useful way.

Long-term memory added the most: +5.6 percentage points. Giving the agent a persistent record of past failures and solutions meant it stopped repeating the same mistakes across sessions.

Tool descriptions and implementations came next: +3.3 points. Better tool definitions reduced hallucinated parameters and format mismatches. This is directly related to the failure catalog from the previous section.

Middleware added +2.2 points. This is the repair layer. The interceptor between model output and tool execution where Awais’s format fixes live.

System prompts, on their own, caused a regression: -2.3 points. Adding more instructions to the system prompt without tuning the other components actually made the agent worse. More prose didn’t help. Better plumbing did.

Build Your Own Repair Layer
The full repair middleware Awais built at CommandCode.ai handles edge cases across dozens of tools. But the core idea fits in about 80 lines of Python. Here’s a minimal version that covers the four most common format violations from the failure catalog.

import json
import re
from typing import Any

def repair_tool_call(tool_name: str, arguments: dict, tool_registry: dict) -> tuple[str, dict]:
    """
    Intercepts a tool call from the model and repairs known format violations
    before the call reaches the execution layer.
    """
    # Repair 1: Normalize tool name case
    # Models like Granite output "Write" when the registry expects "write"
    normalized_name = tool_name.lower()
    if normalized_name not in tool_registry and tool_name not in tool_registry:
        # Try case-insensitive match
        for registered in tool_registry:
            if registered.lower() == normalized_name:
                normalized_name = registered
                break
    else:
        normalized_name = tool_name if tool_name in tool_registry else normalized_name
    # Get the expected schema for this tool
    schema = tool_registry.get(normalized_name, {}).get("parameters", {})
    required_fields = schema.get("required", [])
    properties = schema.get("properties", {})
    repaired_args = {}
    for key, value in arguments.items():
        prop_schema = properties.get(key, {})
        expected_type = prop_schema.get("type")
        # Repair 2: Strip null optional fields
        # DeepSeek sends {"file_path": "/src/main.py", "encoding": null}
        # when the tool expects {"file_path": "/src/main.py"} (encoding omitted)
        if value is None and key not in required_fields:
            continue  # Drop it entirely
        # Repair 3: Wrap bare strings in arrays
        # Model sends "foo" when schema expects ["foo"]
        if expected_type == "array" and isinstance(value, str):
            # Check if it's a string-wrapped array like "[\"foo\", \"bar\"]"
            if value.startswith("["):
                try:
                    parsed = json.loads(value)
                    if isinstance(parsed, list):
                        value = parsed  # Successfully unwrapped
                except json.JSONDecodeError:
                    value = [value]  # Treat as single-item array
            else:
                value = [value]  # Bare string → single-item array
        # Repair 4: Parse string-encoded objects
        # Model sends '{"key": "val"}' as a string instead of an object
        if expected_type == "object" and isinstance(value, str):
            try:
                value = json.loads(value)
            except json.JSONDecodeError:
                pass  # Keep the string if it's not valid JSON
        repaired_args[key] = value
    return normalized_name, repaired_args
That’s the core loop. For each tool call, it normalizes the name, strips null optionals, wraps bare strings into arrays, and parses string-encoded objects. The model’s reasoning about what to do stays untouched. Only the format gets corrected.

One caveat worth being honest about: this only covers categories 1 through 3 from the failure catalog. When the model picks the wrong tool entirely, calls tools in the wrong sequence, or hallucinates a parameter name that doesn’t exist, no amount of format repair saves you. Those are reasoning failures, not wire-format failures, and they still require a better model or better prompting.

You wire it in between the model’s output parser and your tool execution layer:

def execute_with_repair(model_output, tool_registry):
    tool_name = model_output["function"]["name"]
    arguments = json.loads(model_output["function"]["arguments"])
    # Repair before execution
    tool_name, arguments = repair_tool_call(tool_name, arguments, tool_registry)
    # Now execute the tool with cleaned inputs
    return tool_registry[tool_name]["handler"](**arguments)
The pattern is the same one Awais described: intercept, match, repair, forward. The middleware doesn’t need to understand the model’s intent. It just needs to know what format the tool expects and fix the shape when it doesn’t match.

One thing worth noting: this approach works precisely because the failure set is finite. You’re not building a general-purpose error handler. You’re cataloging the specific ways models mangle tool-call formats and writing a targeted fix for each one. When a new model introduces a new failure pattern, you add one more repair rule. The catalog grows, but slowly.

The Proof in the Wild
This isn’t just benchmark data and middleware theory. Real projects are already building on the harness thesis, and the results tell a consistent story.

DeepClaude ripped the model out of Claude Code and replaced it with DeepSeek. The project is a proxy that intercepts Claude Code’s API calls and routes them to DeepSeek V4 Pro. The entire agent loop stays intact: file reading, editing, bash execution, multi-step tool iteration, sub-agent spawning. Only the reasoning engine changes. The cost drops from $25 per million tokens to $0.87. That’s roughly a 29x cost reduction, replicated at the infrastructure level.

And as of May 24, DeepSeek made that price permanent. GPT-5 sits at $2.50/$10 per million input/output tokens. Opus 4.7 at $5/$25.

The project’s own documentation is honest about the limits: no image input, no parallel tool calls, and “on hard reasoning tasks, Claude Opus is still stronger.” But for typical coding work, the harness carries most of the functional load. Swap the model, keep the harness, and the experience stays largely the same.

DeepSeek-TUI took the opposite approach. Instead of adapting an existing harness to a new model, it built a harness from scratch around DeepSeek V4’s native tool-calling format. No repair middleware needed because the harness speaks DeepSeek’s language natively. It hit 5,000 GitHub stars in its first week on trending. The demand for model-specific harnesses is real.

Then there’s Cursor. Same commodity models as everyone else. GPT-5.5, Claude Opus, nothing proprietary. Yet the company crossed $500 million in annualized revenue by mid-2025 and is reportedly in talks at a $30 billion valuation. The moat isn’t the model. Every competitor has access to the same weights. The moat is the harness: how Cursor manages context, sequences tool calls, handles file edits, and recovers from errors.

The market is pricing harness engineering at $30 billion. That’s not a trend. That’s a verdict.
